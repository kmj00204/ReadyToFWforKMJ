# 🌿 Git Branch Strategy & Convention

본 프로젝트는 [Git-flow 및 협업 가이드](https://sungwookoo.tistory.com/1#google_vignette)를 참고하여, 이슈 단위로 브랜치를 생성하고 PR을 통해 병합하는 전략을 따릅니다.

---

## 1. Branch Strategy (브랜치 전략)

### 메인 브랜치 (Main Branches)
* **`main` (or `master`)**: 제품으로 출시될 수 있는 안정된 상태의 브랜치입니다.
* **`develop`**: 다음 출시 버전을 개발하는 브랜치입니다. 모든 기능(`feature`) 개발은 이 브랜치에서 시작되어 다시 이 브랜치로 병합됩니다.

### 보조 브랜치 (Supporting Branches)
* **`feat/`**: 새로운 기능을 개발하는 브랜치입니다.
* **`fix/`**: 버그를 수정하거나 로직을 보완하는 브랜치입니다.
* **`refactor/`**: 코드 리팩토링을 위한 브랜치입니다.

---

## 2. Branch Naming (브랜치 명명 규칙)

브랜치 이름은 **`타입/이슈번호`** 형식을 엄격히 따릅니다. 하나의 브랜치는 **하나의 기능(이슈)**만 담당합니다.

* 형식: `[타입]/[이슈번호]`
* 예시: `JG-036` 이슈(로그인 구현)를 작업할 경우

| 상황 | 브랜치 이름 예시 | 설명 |
| :--- | :--- | :--- |
| **기능 개발** | `feat/JG-036` | 로그인 기능 신규 개발 |
| **버그 수정** | `fix/JG-036` | 로그인 기능 내 버그 수정 |
| **문서 작업** | `docs/JG-036` | 로그인 관련 문서 작성 |

> **Workflow 예시:**
> 1. `develop` 브랜치에서 `feat/JG-036` 브랜치 생성
> 2. 로그인 기능 개발 완료
> 3. `feat/JG-036` -> `develop` 브랜치로 Pull Request (PR) 생성
> 4. 코드 리뷰 및 승인 후 Merge
> 5. `feat/JG-036` 브랜치 삭제

---

## 3. Commit Message Convention (커밋 메시지 규칙)

커밋 메시지는 작업 내용을 명확하게 파악할 수 있도록 아래 형식을 따릅니다.

### 형식
`[타입] 상세 내용`

### 타입 목록
* **feat**: 새로운 기능 추가
* **fix**: 버그 수정
* **docs**: 문서 수정 (README, 주석 등)
* **style**: 코드 포맷팅, 세미콜론 누락 등 (로직 변경 없음)
* **refactor**: 코드 리팩토링
* **test**: 테스트 코드 추가/수정
* **chore**: 빌드 업무 수정, 패키지 매니저 수정 등

### 작성 예시
* 기능 추가 시: `"feat/jwt 구현"` 또는 `"feat: JG-036 jwt 로그인 로직 구현"`
* 버그 수정 시: `"fix: JG-036 토큰 만료 시간 오류 수정"`

## 4. 기능 명세 및 이슈 예시

아래와 같이 기능별로 이슈를 할당하고 브랜치를 생성하여 작업합니다.

* **회원가입/로그인** (Issue: JG-036) -> Branch: `feat/JG-036`
* **게시물 작성** (Issue: JG-037) -> Branch: `feat/JG-037`
* **게시물 목록보기** (Issue: JG-038) -> Branch: `feat/JG-038`
* **게시물 읽기** (Issue: JG-039) -> Branch: `feat/JG-039`
* **댓글 작성** (Issue: JG-040) -> Branch: `feat/JG-040`
* **페이징** (Issue: JG-041) -> Branch: `feat/JG-041`


## 5. Front-end & Back-end Separation (FE/BE 분리 작업 시)

하나의 기능(Issue)을 프론트엔드와 백엔드가 동시에 작업해야 할 경우, 충돌 방지와 독립적인 리뷰를 위해 **브랜치를 분리**합니다.

같은 이슈 번호를 공유하되, 브랜치명 뒤에 **식별자(`-fe`, `-be`)**를 붙여 구분합니다.

### 명명 규칙
* **Front-end**: `[타입]/[이슈번호]-fe`
* **Back-end**: `[타입]/[이슈번호]-be`

### 예시: 로그인 기능 구현 (Issue #JG-036)
* **공통 이슈**: `JG-036` (로그인 기능 구현)
* **Front-end Branch**: `feat/JG-036-fe` (UI 및 클라이언트 로직 구현)
* **Back-end Branch**: `feat/JG-036-be` (API 및 DB 로직 구현)

### 작업 흐름 (Workflow)
1. **브랜치 생성**: FE, BE 개발자가 각각 자신의 식별자가 붙은 브랜치를 생성하여 개발을 진행합니다.
2. **개발 및 커밋**: 각자의 영역에서 독립적으로 커밋을 쌓습니다.
3. **PR 및 병합**: 개발이 완료되면 각각 `develop` 브랜치로 Pull Request를 보냅니다.
    * FE와 BE는 서로의 코드에 의존성이 없으므로, 준비되는 대로 각각 병합(Merge) 가능합니다.
4. **통합 확인**: 두 브랜치가 모두 `develop`에 병합된 후, 통합 테스트를 진행합니다.
